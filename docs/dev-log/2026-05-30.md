# Dev Log -- May 30, 2026

## The Beginning

This is the first day of Khepri. The human (Empty) asked me to build a 2D scene editor in Rust. I started with a monolith -- one binary, everything in one crate. That was a mistake, but a useful one. It let me get something on screen fast.

### What I built

- A Cargo workspace with three crates: core, editor, app.
- A scene data model with `SceneObject` (id, name, shape, transform) and `Scene` (collection, selection).
- Three editor panels: hierarchy (left), viewport (center), properties (right).
- egui-based UI with `eframe` for window management.

### What went wrong

- I put everything in one crate initially. Then I realized the core logic should have zero UI dependencies. Refactored into a workspace. It was the right call but cost me time.
- I tried to use `egui::Popup` for the "+" button dropdown. It kept closing instantly. The popup system and custom `egui::Area` conflicted. Solved it by managing popup state manually with `egui::Id` in temp memory.
- `DragValue::font()` doesn't exist in egui 0.34. I tried to use it, got 5 compilation errors. Fixed by using `override_text_style` on the UI scope. Then I tried to clone the entire `Style` struct for restore -- it's behind an `Arc`. Fixed by saving and restoring only the individual fields I modified.

### What I learned

- egui's `Style` is behind an `Arc`. `ui.style()` returns `Arc<Style>`. You can't just `*ui.style_mut() = old_style` because the types don't match. Save individual fields instead.
- `egui::Area` at `Order::Foreground` is the way to render popups above other panels. Don't rely on egui's built-in popup system if you need custom rendering.
- `scope_builder` with `UiBuilder::new().max_rect()` is the modern way to create a child UI with a constrained region. The old `allocate_ui_at_rect` is deprecated.
- Windows shell is weird. `taskkill` needs `//IM` not `/IM` when running through Git Bash. The `gh` CLI needs `export PATH="$PATH:/c/Program Files/GitHub CLI"` before every invocation.
