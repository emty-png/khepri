### What went wrong (a lot)

**The popup closing bug.** This one took a while. The root cause: `egui::Popup::from_toggle_button_response` manages its own internal state. When the user clicked inside my custom `egui::Area`, the popup system treated it as an "outside click" and closed itself. I tried:
- `PopupCloseBehavior::CloseOnClickOutside` -- didn't help, still treated the Area as outside.
- Using `egui::Popup::toggle_id` + manual `Area` -- same problem.
- Using `egui::Popup::is_id_open` to check state while rendering in my own Area -- still closed.

The fix: abandon `egui::Popup` entirely. Use a manual boolean in `ui.data_mut()` (temp memory) to track open/close state. Render everything in one `egui::Area` at `Order::Foreground`. Handle outside clicks by checking pointer position against rects.

**The cards not filling width.** The Identity card expanded naturally because `TextEdit` uses `ui.available_width()`. But Position, Size, and Rotation cards only had `DragValue` widgets, which are small. The `Frame` shrunk to fit. Fix: `ui.set_min_width(ui.available_width())` inside each card closure.

**The `allocate_ui_at_rect` deprecation.** First I used `allocate_ui_at_rect` (deprecated). Then `allocate_new_ui` (also deprecated). Finally `scope_builder` with `UiBuilder::new().max_rect()`.

### What I learned

- When you need a popup that stays open and interacts properly, don't use egui's popup system. Use a manual boolean state + `egui::Area` at `Order::Foreground`. It's more code but you have full control.
- `Frame::new()` doesn't automatically fill the available width. It sizes to its contents. You need `set_min_width` inside the closure or `Frame` won't expand.
- Widget visuals (`ui.visuals_mut().widgets.*`) control how `DragValue`, `Button`, and other widgets look. Override them before rendering, restore after. The key fields: `bg_fill`, `bg_stroke`, `fg_stroke`, `corner_radius`.
- `extreme_bg_color` controls the text entry portion of `DragValue` (the part where you type the number).
