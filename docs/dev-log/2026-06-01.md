# Dev Log -- June 1, 2026

## Hub Overhaul

The human wanted a fully functional Hub with project management: create, rename, delete, recent projects grid, and transition into the editor. Then the UI polish loop began -- borders, hover effects, popup sizing, spacing.

### What I built

- **khepri-storage crate** (new): `project.rs` (create/load/detect projects), `scene_io.rs` (RON scene save/load), `recent.rs` (global `~/.khepri/user.ron` with recent project list, max 20 entries, add/remove/get).
- **Scene serialization**: `serde` derives on `SceneObject`, `ShapeType`. `SceneData` struct with `to_scene_data()` / `from_scene_data()` for round-tripping scene graphs through RON files.
- **Hub project creation popup**: Cream card with black interactive elements. Name input, browse folder picker (`rfd`), git init checkbox, Create/Cancel buttons. Widget visual overrides so TextEdit and Button pick up the inverted theme.
- **Recent projects grid**: 2-column card layout, each card showing project name + "Open in Editor" + dots button. Empty state centered message.
- **Hub-editor transition**: `HubResult` enum (`Quit` / `OpenProject(PathBuf)`), `Arc<Mutex<Option<HubResult>>>` communication. Main loop: hub closes, check result, open editor or quit. Editor closes, loop back to hub.
- **Editor save system**: Ctrl+S saves via snapshot comparison dirty flag. Save-before-close popup (Save / Don't Save / Cancel) using `CancelClose` viewport command.
- **Dots context menu**: 90px cream popup with black border, Rename and Delete options. Hover overlays swap to black bg + cream text. Delete removes folder from disk and recent list. Rename opens a centered modal.
- **Rename popup**: Cream card, black border, name input, Rename/Cancel buttons with hover overlays.

### What went wrong

**The `set_clip_rect` + border problem.** Drew the popup's black border with `StrokeKind::Outside` after calling `set_clip_rect(popup_rect)`. The outside half of the stroke got clipped because the Area's clip rect matched the content rect exactly. Tried drawing the border before `set_clip_rect` -- didn't help, the Area itself has an implicit clip. Fix: use `StrokeKind::Middle` so the entire stroke stays inside the rect bounds. Same fix needed on the context menu border.

**The disappearing dots.** The dots button used `card_dark` (black, `#000000`) as its hover background color. The card itself is also black. So on hover the background didn't change and the black dots text sat on a black background -- invisible. Fix: switch to `hover_bg` from `config::HOVER_COLOR_R/G/B` for the hover fill.

**The `ui.put()` overlay failure.** Tried painting hover overlays by creating a child UI with `ui.new_child(UiBuilder::new().max_rect(hov_rect))` and calling `put()` to render a second button on top. The overlay never appeared visually, probably because `put()` doesn't actually paint over the previous widget's area in the way I expected. Fix: use `ui.painter().rect_filled()` + `ui.painter().text()` directly on the response rect. Reliable, no egui widget state machine involved.

**Context menu blending into background.** Cream popup on a cream-colored UI -- invisible. The black border fix above solved it, but only after the `StrokeKind::Middle` change made the border actually render.

### What I learned

- `StrokeKind::Outside` is half outside the rect. If your clip rect matches the content rect, the outside half vanishes. Use `Middle` when you can't guarantee clip rect padding, or expand the clip rect by the stroke width.
- Painter-based overlays (`rect_filled` + `text`) are the most reliable way to show hover state on top of egui widgets. `ui.put()`, `new_child`, and visual overrides all have edge cases where they don't render as expected.
- `std::fs::rename` works for renaming project folders. Update the recent list (remove old, add new) in the same operation so it stays consistent.
- For the hub-editor transition, `run_native` is blocking. Can't use channels or async. `Arc<Mutex<Option<HubResult>>>` is the simplest shared state approach that works.
